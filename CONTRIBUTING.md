# 参与贡献 / Contributing

感谢你愿意改进本项目。请先读完本页再动手——本项目对**代码风格**和**提交方式**
有硬性约定,不符合的提交会被退回。

---

## 一、代码风格:以 Vles0123 原始版本为准

本项目的代码风格**以原作者 Vles0123 提供的初版(仓库 git 历史中的 `baseline` 提交)为唯一基准**。
后续所有提交都必须延续这套风格,不得擅自引入新的范式、框架或工具链。具体约定如下:

1. **只用 Python 3 标准库写核心。**
   后端 `main.py` 仅依赖标准库(`http.server` / `socketserver` / `sqlite3` /
   `json` / `csv` / `threading` / `datetime` 等),**不引入 Web 框架**(不要 Flask /
   FastAPI / Django)。确有必要的可选依赖,必须用 try/except 兜底,缺失时能降级运行
   (参考现有对 Redis、argon2、pypinyin 的处理方式)。

2. **单文件后端 + 独立 HTML 前端,无构建步骤。**
   后端逻辑集中在 `main.py`;前端是可直接打开的 `.html`,**不引入打包器 / 编译步骤**
   (不要 npm / webpack / TypeScript / React)。CSS 写在 `app.css`,字体自托管在 `fonts/`。

3. **4 空格缩进。** 不使用 Tab,不使用 2 空格。

4. **中文注释,逐段解释意图。**
   沿用初版的注释习惯:在关键逻辑前用中文注释说明"这一段在做什么、为什么",
   而不是只翻译代码。注释是给下一位中学老师 / 学生看的。

5. **命名与结构沿用初版:**
   - 函数、变量用 `snake_case`;有状态的组件用类封装(如连接池),请求处理用函数。
   - 数据库访问直接用 `sqlite3` 写内联 SQL,不引入 ORM。
   - 保持"路由表 + 处理函数"的组织方式,新增接口在现有表里登记,不另起架构。

6. **不提交运行产物与数据。**
   `club_system.db`、日志、`__pycache__`、Rust `target/` 等已在 `.gitignore` 内,
   任何学生数据、密码、本地缓存都**绝不入库**。

> 一句话:看 `baseline` 提交里 Vles0123 怎么写,你就怎么写。拿不准时,模仿现有代码,
> 而不是按你自己习惯的写法另起一套。

---

## 二、提交方式:只走 Pull Request,禁止直推 main

**所有改动必须通过 Pull Request 合入。直接向 `main` 分支推送的提交一律不受理**
(`main` 已开启分支保护,直推会被拒绝)。

标准流程:

1. Fork 本仓库,或在本仓库新建分支(分支名形如 `fix/xxx`、`feature/xxx`)。
2. 在你的分支上提交改动,提交信息写清楚"改了什么、为什么"。
3. 推送分支,发起 **Pull Request**,目标分支选 `main`。
4. 等待维护者(BIRI GA / KeikaJames)review;按意见改完再请求合并。
5. 通过 review 后,由维护者合并。

PR 自检清单(发起前请逐条确认):

- [ ] 符合上面第一节的全部风格约定(stdlib、单文件、4 空格、中文注释、snake_case)。
- [ ] 没有引入新的重依赖 / 框架 / 构建步骤;可选依赖有 fallback。
- [ ] 没有提交任何数据文件、密码或运行产物。
- [ ] 本地能跑起来(`python3 main.py` 可启动,功能正常)。
- [ ] 提交信息清晰,一个 PR 只做一件相对完整的事。
